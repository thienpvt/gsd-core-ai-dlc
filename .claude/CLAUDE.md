## Project

**GSD Governance Overlay (AI-DLC × GSD Core)**

A working extension to GSD Core that layers AI-DLC-style enterprise SDLC governance onto GSD's long-running development runtime — without polluting the context window. GSD Core stays the runtime brain (`.planning/`, roadmap, state, phase/execution loop); AI-DLC contributes a governance overlay delivered as indexed, on-demand **rule packs**. Instead of injecting all steering markdown into every request, a selection engine picks only the rules relevant to the current task and phase and injects short summaries, loading full rule detail only when truly needed. It is for teams who want AI-driven development at scale while preserving enterprise compliance, auditability, and human approval gates.

**Core Value:** The rule selection engine correctly injects only the relevant AI-DLC rule summaries for the current task and phase — enough governance to be safe, little enough to avoid context bloat. If this fails, the entire premise fails.

### Constraints

- **Architecture**: Overlay on GSD Core, not a fork — must hook GSD's existing discuss/plan/execute/verify/ship loop cleanly and stay upgrade-safe
- **Context budget**: Per-request injection must be summaries-only; full rule bodies load on demand — this is the defining constraint, not a nice-to-have
- **Enforcement**: Markdown is advisory; binding enforcement must route through CI/CD, SAST, tests, policy-as-code, and human approval via tool-agnostic contracts
- **Portability**: Gate contracts and audit schemas must be engine-neutral with adapter stubs — no hard dependency on any specific policy/scan vendor
- **Auditability**: Every governed task must be able to produce a complete, review-ready audit artifact

## Grounding: What Was Actually Observed

- npm package `@opengsd/gsd-core`, version `1.7.0-rc.2` (latest stable release line 1.6.1).
- `engines`: Node `>=22.0.0`, npm `>=10.0.0`.
- **CommonJS** — no `"type": "module"`; `bin` entries are `.cjs`/plain JS. `gsd-tools` → `gsd-core/bin/gsd-tools.cjs`; `gsd-core` → `bin/install.js`; also `gsd_run` and `gsd-mcp-server`.
- Runtime deps are minimal: `@anthropic-ai/claude-agent-sdk ^0.2.84`, `ws ^8.21.0`, optional `fallow`.
- **`js-yaml` is already a dependency** (listed under devDependencies alongside `globals`).
- Build = `tsc` only (`tsc -p tsconfig.build.json`), **no bundler**. Plugin skills, capability registry, hooks, and a "loop-host contract" are generated by custom node scripts under `scripts/`.
- Tests = custom runner (`node scripts/run-tests.cjs`, suites: unit/integration/install/security/slow), coverage via `c8 ^11`, mutation via `@stryker-mutator/core ^9.6.1`, property tests via `fast-check ^4.8.0`. TypeScript `^6.0.3`, ESLint `^9.39.4`, typescript-eslint `^8.60`.
- Extension surfaces observed as directories: `commands/gsd/`, `agents/`, `skills/`, `capabilities/`, `hooks/`, `.claude-plugin/`, `.opencode/plugins/`. README states the **installer is required for cross-runtime compatibility — do not copy files from `agents/`/`commands/` directly** (inferred implication: extend through the installed surfaces, not by forking).
- 5-step loop confirmed: Discuss → Plan → Execute → Verify → Ship. `CONTEXT.md` + `STATE.md` are the artifacts that survive session boundaries.
- *Not directly visible from the web view:* exact contents of `commands/gsd/`, the internal API of `gsd-tools.cjs`, and whether a root `config.json` exists (README/docs reference `docs/CONFIGURATION.md`). These are marked inferred where used below.
- `aidlc-rules/` is the release artifact, split into **`aws-aidlc-rules/` (core workflow rules)** and **`aws-aidlc-rule-details/` (detailed rules *conditionally referenced* by the core rules)**. This core-vs-detail split is exactly the summary/lazy-detail separation this project needs — the source corpus already models it.
- Rules are **plain Markdown (`.md`)**. There is **no YAML frontmatter in the rules themselves**; frontmatter appears only in the Cursor `.mdc` wrapper (`description`, `alwaysApply: true`).
- Authoring convention: rule files use `## Rule <PREFIX-NN>: <Title>` headings, each with a **Rule** section (the requirement) and a **Verification** section (checks the model evaluates). Rules are "blocking by default." Opt-in extensions use a `<name>.opt-in.md` multiple-choice companion file.
- Phases = **Inception (WHAT/WHY) → Construction (HOW) → Operations (deploy/monitor, future)**, mapped to subdirs under `aws-aidlc-rule-details/` (`common/`, `inception/`, `construction/`, `extensions/{security,testing,resiliency}`, `operations/`). Workflow is "adaptive" — only runs stages that add value.
- The repo ships real enforcement config: `.bandit`, `.checkov.yaml`, `.semgrepignore`, `.grype.yaml`, gitleaks — a concrete signal of which SAST/scan tools the AI-DLC model expects downstream (useful for naming adapter stubs).

## Recommended Stack

### Core Technologies

| Technology | Version | Purpose | Why Recommended |
|------------|---------|---------|-----------------|
| Node.js | `>=22.0.0` | Runtime | Matches GSD Core's `engines` exactly. Staying in-runtime is the difference between a native extension and a second toolchain the installer can't manage. A Python/Go engine would need its own packaging, break `npx @opengsd/gsd-core` distribution, and can't reuse `gsd-tools.cjs`/loop-host contracts. |
| TypeScript | `^6.0.3` | Implementation language | GSD Core is authored in TS and compiled with `tsc`. Matching the version and `tsconfig` conventions keeps the extension buildable by GSD's existing pipeline with zero new build tooling. |
| CommonJS modules | n/a | Module format | GSD Core is CJS (`.cjs` bin shims, no `"type":"module"`). Authoring the extension as CJS lets `gsd-tools.cjs` and generated node scripts `require()` it without ESM interop friction. |
| Markdown + YAML frontmatter | n/a | Rule-pack authoring format | Bodies stay Markdown (human-authored, matches AI-DLC's existing corpus and `## Rule NN:` + Verification structure). Frontmatter adds the machine-selectable metadata AI-DLC lacks: `id`, `scope`, `triggers`, `phases`, `severity`, `summary`, `detailPath`. Mirrors the `.mdc` frontmatter convention AI-DLC already uses for Cursor. |
| JSON Schema (draft 2020-12) | n/a | Gate contracts + audit schemas | The tool-agnostic contract layer. A schema is engine-neutral: any CI/SAST/policy engine can be wrapped to produce/consume schema-valid JSON. This is the concrete alternative to hard-coding OPA/Rego or GitHub Actions. |
| Ajv | `8.20.0` | JSON Schema validator | De-facto standard Node validator, supports draft 2020-12 (`ajv/dist/2020`). Validates gate results and audit artifacts at runtime so adapters can't silently emit malformed records. Zero-runtime-dep, CJS-compatible. |

## Gate Contracts + Audit Schemas — How to Represent Them (prescriptive)

- Define **JSON Schema (draft 2020-12)** documents for: (a) the gate-request contract (what a gate needs), (b) the gate-result contract (normalized pass/fail/waived + findings), and (c) the audit-artifact schema (requirements covered, rules applied, rules skipped + reasons, tests executed, remaining risks, approvals required).
- Validate every adapter's output with **Ajv** at runtime. Malformed = hard fail, so no adapter can corrupt the audit trail.
- **Adapter stub pattern:** define a single `GateAdapter` TypeScript interface (`evaluate(request): Promise<GateResult>`), ship reference **no-op/echo stubs**, and translate tool output → schema-valid `GateResult`. Name the reference stubs after tools AI-DLC already implies — semgrep, bandit, checkov, grype, gitleaks (SAST/scan), a generic exit-code CI adapter, and a human-approval adapter — but ship them as stubs, not first-class integrations.

## Alternatives Considered

| Recommended | Alternative | When to Use Alternative |
|-------------|-------------|-------------------------|
| Node.js/TS native extension | Separate Python/Go service | Only if selection needed heavy ML/embeddings at scale — it doesn't. A separate runtime breaks GSD's `npx` distribution and installer model. |
| gray-matter | Parse frontmatter manually with GSD's `js-yaml` | If you want zero new runtime deps and full control of parsing; slightly more code, avoids gray-matter's transitive `js-yaml@3`. |
| Markdown + YAML frontmatter | Pure JSON or pure YAML rule files | If rules were machine-generated only. Rejected: AI-DLC rules are human-authored prose with `## Rule NN:`/Verification structure; markdown bodies preserve that, frontmatter adds machine-selectability. |
| picomatch | minimatch | If team prefers the familiar minimatch API. Functionally equivalent for scope globs; picomatch is lighter/faster. |
| Deterministic trigger matching | Embeddings/vector search | Only for very large, unstructured, cross-lingual corpora where keyword triggers genuinely fail. Not this project — and it forfeits determinism + auditability. |
| Ajv (draft 2020-12) | Zod | If contracts were TS-internal only. Rejected as the *contract* layer: the tool-agnostic boundary must be a language-neutral schema other engines can read; JSON Schema is that lingua franca. Zod is fine for internal TS validation if desired, but the published contract stays JSON Schema. |
| node:test + c8 | jest / vitest | If the team already standardizes on one. Rejected to match GSD Core's zero-extra-dep custom runner model. |

## What NOT to Use

| Avoid | Why | Use Instead |
|-------|-----|-------------|
| OPA/Rego as a hard dependency | Directly violates the tool-agnostic + no-lock-in constraint; forces a policy runtime on every consumer | JSON Schema gate contracts + `GateAdapter` stubs (an OPA adapter can be one optional stub) |
| GitHub Actions (or any single CI) as the enforcement mechanism | Vendor lock-in; the overlay must run under any CI/none | Engine-neutral gate-result schema + a generic exit-code CI adapter stub |
| Embeddings / vector DB for selection | Non-deterministic, unexplainable (breaks auditability), heavy dep + index lifecycle, overkill for a bounded human-authored corpus | Deterministic keyword/glob trigger + scope matching (picomatch) |
| Forking / patching GSD Core internals | Breaks upstream upgradability — explicit Out of Scope; README warns against copying `agents/`/`commands/` directly | Extend via installed surfaces (`commands/`, `skills/`, `capabilities/`, `hooks/`) and hook the discuss/plan/execute/verify/ship loop |
| A bundler (webpack/esbuild/rollup) | GSD ships none; a CLI extension gains nothing and diverges from GSD's `tsc`-only build | Plain `tsc` with GSD's `tsconfig` conventions |
| ESM-only authoring | GSD Core is CommonJS; ESM-only would need interop shims into `gsd-tools.cjs` | CommonJS to match the host runtime |
| Copying full AI-DLC steering into context | The exact anti-pattern this project exists to kill | Frontmatter index → summary injection → lazy `detailPath` load |
| Injecting `year` into web/version queries | Biases toward stale dated content | Query without year; verify against npm registry / official docs (done here) |

## Stack Patterns by Variant

- Use the deterministic trigger/scope matcher with a generated `rule-index.json`.
- Because it is explainable, cheap to load, and every decision is audit-recordable.
- Add it as a `GateAdapter` stub implementation, not a core dependency.
- Because the JSON Schema contract keeps the core engine-neutral; adapters are swappable at the edge.
- Add Fuse.js `7.4.2` as an opt-in author-time tool only, never in the runtime selection path.
- Because determinism must be preserved for audit; fuzziness at authoring time is safe.
