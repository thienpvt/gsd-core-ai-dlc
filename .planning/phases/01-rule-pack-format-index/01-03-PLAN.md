<objective>
Add scope organization and precedence resolution (PACK-02) on top of the 01-01 walking skeleton: derive a rule's scope from its directory tier (D-09/D-10), resolve same-id collisions across enterprise/domain/project by defined precedence (project > domain > enterprise), and reject ambiguous or mis-scoped rules at build time. This is the deterministic business logic the whole governance overlay keys off — a wrong winner or a silently-accepted mis-scoped rule is a governance-bypass, so the resolution logic is built test-first.

Implements directory-as-source-of-truth scope (D-09: frontmatter scope must match the tier it physically lives under, mismatch rejected), the three-tier store layout under aidlc-rules/ (D-10), full-replacement cross-tier override with the loser recorded as superseded (D-11), and duplicate-id rules: globally-unique slugs where a same-tier duplicate is a hard error and a cross-tier duplicate is the override signal (D-12). Task 3 wires the resolver into buildIndex so the emitted rule-index.json carries the correct winner plus superseded provenance without ever leaking a body (PACK-04 preserved).

Wave/dependency justification: this plan builds only on the 01-01 contract (loadRules, ParsedRule, the src/types.ts type set including Scope/SupersededRecord/RuleIndexRecord, and the buildIndex(rootDir) seam). Precedence and scope resolution are orthogonal to 01-02's schema hardening — they operate on the id and scope fields present in 01-01's minimal schema, and this plan's fixtures use only minimal-schema-valid frontmatter. It therefore depends_on 01-01 alone and runs in wave 2 in parallel with 01-02 (zero files_modified overlap between the two). It touches src/index/build.ts, which 01-01 also creates, so it is correctly a wave after 01-01.

Purpose: Correct precedence is the difference between a governance rule applying and being silently dropped; scope-vs-directory consistency stops a rule claiming authority its location does not grant.
Output: src/rules/scope.ts (deriveScope, ORDINAL, assertScopeMatchesDirectory, resolvePrecedence, ResolvedRule) + a table-driven unit suite + buildIndex integration + a fixture-backed precedence/mismatch integration test.
</objective>
