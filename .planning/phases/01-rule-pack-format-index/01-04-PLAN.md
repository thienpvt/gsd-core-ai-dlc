<objective>
Capstone the phase's PACK-04 guarantee: turn the walking skeleton's by-construction "no body in the index" promise into an enforced, machine-proven invariant, and surface the PACK-03 binding-without-contract rejection at index-build time. 01-01 builds the index by field whitelist and 01-03 preserves that through precedence, but neither ships the output schema RESEARCH calls the body-leak guard (01-RESEARCH.md ~line 420) nor a property test — so PACK-04 is currently only example-tested. This plan adds the definitive proof.

It adds three guards on the emitted rule-index.json: (1) a draft 2020-12 output schema (src/schema/rule-index.schema.json) whose per-rule additionalProperties:false leaves no place for a body/content key (D-05); (2) validateIndex, wired into buildIndex so any stray key aborts the build; and (3) a fast-check property that generates arbitrary corpora with per-rule-unique body canaries and asserts none ever reach the serialized index. It also proves ROADMAP success criterion 3 at the BUILD level (binding-without-enforcement fails buildIndex, not just validateFrontmatter — D-15/PACK-03) and success criterion 4 (a detailPath-bearing rule's pointer is carried verbatim into the index).

Scope fence (explicit, so the boundary is auditable): this plan covers only CARRYING the detailPath pointer through the index. It does NOT resolve, traverse, open, or existence-check the target file — that is D-06/D-07/D-08, tagged [deferred] to Phase 3 in 01-CONTEXT.md. No Phase 1 code dereferences detailPath.

Wave/dependency justification: this is wave 3 with depends_on ["01-01","01-02","01-03"] because it needs all three seams at once — the buildIndex/writeIndex CLI seam and whitelist from 01-01, the hardened frontmatter schema from 01-02 (the binding->enforcement if/then that makes the build-time binding fixture fail), and the superseded records from 01-03 (so the output schema must cover superseded[] shapes). It modifies src/index/build.ts, which 01-01 and 01-03 also touch, so it must run in a strictly later wave than both. Addresses review concern: "Missing 01-04 (index no-body hardening)" and "PACK-04 claimed but only example-tested".

Purpose: PACK-04 is a load-bearing guarantee every later phase inherits (Phase 2 selects over the index, Phase 3 injects summaries, Phase 5 audits from it) — a leaked body defeats the entire anti-context-bloat premise, so the guarantee must be schema-enforced and property-proven, not construction-only.
Output: rule-index.schema.json + validateIndex (wired into buildIndex) + a fast-check no-body property test + a build-time binding-rejection and detailPath-pointer integration suite with POSIX pointer assertions.
</objective>
