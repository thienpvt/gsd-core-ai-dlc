Structure (copy from validate-signal.ts, swap schema + target type):
- import Ajv2020, { type ValidateFunction } from "ajv/dist/2020.js"
- import addFormats from "ajv-formats"
- import schema from "../schema/gate-result.schema.json" (no import attribute — resolveJsonModule handles it; import attributes illegal on CJS require())
- import type { GateResult } from "./types.js"
- const ajv = new Ajv2020({ allErrors: true, strict: true, strictRequired: false }) — match validate.ts's options (strictRequired:false because gate-result may grow if/then branches later; harmless now)
- addFormats(ajv) — enables the date-time format check on evaluatedAt
- If the x-binding keyword triggers a strict-mode rejection (discovered in 07-01 Task 3 test), register it: ajv.addKeyword({ keyword: "x-binding", type: "object", schemaType: "string" }) before compile. If 07-01 confirmed x-binding compiles fine without registration, skip this.
- const validate: ValidateFunction = ajv.compile(schema) — compiled once at module load
- Local formatErrors(errors) function: same shape as validate-signal.ts's formatSignalErrors (instancePath || "(root)", message, surface params.missingProperty + params.allowedValues). Returns joined "\n" lines.
- export function validateGateResult(result: unknown): asserts result is GateResult — if !validate(result), throw new Error("invalid gate-result:\n" + formatErrors(validate.errors)). The assertion signature narrows the type for callers (TS assertion function).

The function is pure: no I/O, no clock, no random — same as validate-signal.ts. The only side effect is a throw on malformed input.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| adapter output → audit trail | GateResult crosses from an adapter (untrusted structured output) into the audit trail; a malformed result silently corrupts the record (ENF-02 threat) |
| schema → compiled validator | The gate-result schema is compiled into the validator at module load; a compile failure means NO validation happens (fail-open risk) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-04 | Tampering | validate-gate-result.ts | high | mitigate | TDD: 12 cases pin the validator's behavior — missing required, out-of-enum status, bad ISO-8601, extra properties all hard-fail. RED→GREEN cycle proves the behavior did not pre-exist |
| T-07-05 | Integrity | validateGateResult compile-once | high | mitigate | ajv.compile throws at module load on a schema typo — the test "the validator is compiled once at module load" catches this (a failed import fails the whole test file) |
| T-07-06 | Denial of Service | validateGateResult on huge input | low | accept | Ajv validation is bounded by the schema; no unbounded recursion. Phase 7 stubs produce small outputs; DoS is a Phase 8+ concern when real adapters ship |
| T-07-SC | Tampering | npm installs | low | accept | No new deps — pure stdlib + existing ajv; no install task |
