<objective>
Construct the labeled evaluation set that is the acceptance evidence for the Core Value: a controlled `eval-rules/` corpus plus a labeled `(task, phase, scopeConfig) -> expectedRuleIds` case set, with an integrity test that proves the ground truth is internally consistent before the selection engine exists. This is Wave 1 of Phase 2 and blocks nothing on new code — it reuses Phase 1's `buildIndex` to compile the fixture corpus and validate that every expected rule id is real.

The eval set is deliberately a dedicated fixture store under `test/fixtures/eval/` (NOT the live `aidlc-rules/`) so ground truth stays stable and cannot drift when a production rule is edited (02-CONTEXT.md "eval set & recall gating"). The corpus is engineered so a single controlled signal exercises each selection behavior in isolation: one rule per trigger axis, an exclude carve-out, an empty-triggers always-in-phase critical rule (D-03), an out-of-phase rule, domain-subscription gating, and a project-over-enterprise precedence collision (D-11) that yields a real `superseded` loser.

Purpose: under-injection (a critical rule silently never firing) is the #1 project risk; the Wave-3 recall gate (`criticalRecall === 1.0`) is only meaningful if the ground truth it measures against is verified. This plan produces and validates that ground truth so a mislabeled or typo'd expected id cannot let a missed critical rule hide.
Output: 11 fixture rule files across enterprise/domain/project tiers, one labeled `eval-cases.json` (~12 cases), and `src/select/eval-fixtures.test.ts` proving corpus-builds + expected-ids-resolve + coverage-present.
</objective>
