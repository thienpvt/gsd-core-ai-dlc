    - The caller at line 24 and the usage at lines 120-141 (spawnSync with GSD_TOOLS) — if GSD_TOOLS points to a non-existent file, spawnSync fails with ENOENT, not a clear "gsd-tools not found" message
  </read_first>
  <action>
    Replace the `?? candidates[0] as string` fallback with an explicit handling. Two acceptable forms:
    Form A (throw — preferred for a test helper, makes the failure loud): if no candidate exists, `throw new Error("gsd-tools not found in any candidate location: " + candidates.join(", "))`. This removes the `as string` cast and surfaces the missing-runtime condition immediately.
    Form B (return null + caller guards — more invasive): change the return type to `string | null`, return `null` if none found, and update the caller at line 24 + the usage at lines 115/120 to guard with `if (!GSD_TOOLS) { t.skip("gsd-tools not installed"); return; }`.

    Prefer Form A — the test file already has a `t.skip` guard at line 115 (`if (!existsSync(GSD_TOOLS))`), so if resolveGsdTools throws, that guard never runs and the test fails with a clear "gsd-tools not found" message instead of a confusing ENOENT from spawnSync. BUT: Form A would break the `t.skip` guard at line 115 because resolveGsdTools is called at module-load time (line 24, top-level `const GSD_TOOLS = resolveGsdTools()`), so a throw would crash module import. Therefore use Form B: return `string | null`, update line 24 to `const GSD_TOOLS = resolveGsdTools()` (now `string | null`), keep the existing `if (!existsSync(GSD_TOOLS))` guard at line 115 (which already handles null — `existsSync(null as any)` would throw, so change to `if (!GSD_TOOLS || !existsSync(GSD_TOOLS))`). Remove the `as string` cast entirely.

    Commit: `refactor(06-03): resolveGsdTools handles undefined fallback explicitly`.
