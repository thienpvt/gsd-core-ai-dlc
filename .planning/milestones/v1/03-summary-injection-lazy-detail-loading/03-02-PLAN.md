<objective>
Build the lazy detail loader — the second half of Phase 3's anti-bloat mechanism (SEL-03). `governance rule-detail <id>` fetches exactly one full rule body on demand by id, and this is the ONE sanctioned place in the whole system where a rule body surfaces (index, select, inject all stay body-free). This is Wave 2, depending on 03-01 because both plans modify `src/cli/index.ts` (dispatch) — a shared-file implicit dependency that serializes them.

This plan also lands Phase 1's deferred `detailPath` decisions and resolves the Phase 1 code-review finding IN-05. The load-bearing piece is a single-sourced `resolveDetailPath` resolver that both resolves a detailPath relative to its declaring rule file (D-08) and enforces the traversal guard (IN-05: reject an absolute path or any `..`-escape outside the pack root) BEFORE any file is opened. That one function is used by both build-time validation (D-07, the primary guard extending `buildIndex`) and fetch-time `rule-detail` (the backstop), so the two guards can never drift (Pitfall 8). A summary-only rule with no detailPath returns its summary plus a clear no-detail signal (D-06) — not an error.

Built test-first per the active TDD mode: the resolver + its traversal guard, the build-time validation, and the lazy single-body fetch are business logic with defined I/O, so they are RED-first (a `test(03-...)` commit) then GREEN. The fixtures (a valid detailPath + its target, plus negative fixtures for missing/absolute/escape) are execute-type scaffolding.

Note the cross-phase break this plan owns: Phase 1's `src/index/build-guards.test.ts` currently asserts `buildIndex(DETAILPATH_STORE)` does NOT throw over a fixture whose detail target is intentionally absent. D-07 build-time validation reverses that expectation, so this plan authors the missing target file and updates that stale assertion in the same wave.

Purpose: complete the "index -> select -> inject summaries -> load one body on demand" loop that proves the anti-bloat premise end-to-end.
Output: `resolveDetailPath` + its tests, the D-07 validation in `buildIndex`, the `governance rule-detail` command wired into dispatch with a smoke test, and the detail fixtures.
</objective>
