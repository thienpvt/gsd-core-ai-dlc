<objective>
Build the selection engine core: the pure function `select(index, signal, config)` that reads Phase 1's `rule-index.json` shape and deterministically classifies every candidate rule as selected (with the matched axis + value) or skipped (with a reason from the AUDIT-02-aligned enum). This is the Core Value of the whole project — the component the roadmap flags as riskiest and highest-value — so it is built strictly test-first per the active TDD mode: failing unit + skip-reason tests committed first (RED), then the minimal implementation that turns them green (GREEN).

The engine reuses Phase 1's locked trigger semantics verbatim (D-01 multi-axis, D-02 OR-combine + exclude-wins, D-03 empty=always-in-phase, D-04 per-axis match rules) and the precedence result Phase 1 already computed (D-11 winners carry losers under `superseded[]`). It adds the new selection types alongside the Phase 1 type set (never redefining `RuleIndex`/`Triggers`/etc.), a `TaskSignal` Ajv schema so a malformed input fails loud instead of silently selecting nothing, and the fixed gate-order pipeline (phase -> scope -> trigger) that makes each skip reason deterministic.

Purpose: under-injection — a critical rule silently never firing — is the #1 project risk. A pure, deterministic, fully-observable selector is the concrete guard: every rule is accounted for with a reason, empty triggers are correctly read as always-in-phase (not "never fires"), and a bad input throws rather than quietly returning nothing.
Output: the `TaskSignal`/`SelectionConfig`/`SelectionResult`/`SkipReason`/`SelectedRule`/`SkippedRule` types, `task-signal.schema.json` + `validateSignal`, the pure `select()` core, and its RED-first unit + per-skip-reason test suites. The token-budget flag, recall/precision measurement, fast-check invariants, and CLI wiring are 02-03.
</objective>
