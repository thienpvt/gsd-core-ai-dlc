<objective>
Complete the selection engine: the recall/precision measurement that gates the Core Value, the fast-check invariants that prove the matching core holds across arbitrary corpora, the token-budget estimator with a loud overflow signal (SEL-05), and the `governance select` CLI command that surfaces per-rule selection observability (SEL-04). This is Wave 3, depending on the eval set (02-01) and the pure `select()` core (02-02). Built test-first per the active TDD mode: the recall gate, budget behavior, and invariants are RED-first, then implemented GREEN.

The load-bearing piece is the recall gate. Under-injection (a critical rule silently never firing) is the #1 project risk, so a test asserts `criticalRecall === 1.0` and `highRecall >= 0.9` over the 02-01 labeled set and FAILS THE BUILD otherwise — the single most important test in the milestone. Precision is reported for observability but NEVER gated: gating precision would pressure the engine toward under-injection, which 02-CONTEXT explicitly rejects. The token budget follows the same fail-loud stance: on overflow the engine flags `budgetExceeded` and the CLI exits non-zero, but the engine NEVER truncates a selected rule to fit — dropping a rule could drop a critical one.

Purpose: this plan turns the pure `select()` core into proven, budget-guarded, CLI-invokable governance. The recall gate is the concrete enforcement of "enough governance to be safe"; the budget signal is the concrete enforcement of "little enough to avoid context bloat" — the two halves of the Core Value.
Output: `estimateTokens` + `PER_RULE_OVERHEAD`, the budget computation wired into `select()`, the `eval-harness` + build-gating `recall.test.ts`, the fast-check `select.property.test.ts`, and the `governance select` command wired into the CLI dispatch with a smoke test.
</objective>

## Artifacts This Plan Produces

- `src/select/tokens.ts` — exports `estimateTokens(text: string): number` = `Math.ceil(text.length / 4)` (deterministic, zero-dep, documented as an estimate per 02-CONTEXT token estimator) and `PER_RULE_OVERHEAD` (a single-sourced small integer constant, ~6 tokens, approximating the id/severity framing Phase 3 will wrap each summary in — documented as refinable by Phase 3 per 02-CONTEXT Claude's-discretion).
- `src/select/select.ts` — MODIFIED (additive): after building `selected[]`, compute `used = Σ (estimateTokens(rule.summary) + PER_RULE_OVERHEAD)`; resolve `limit` from `config.budget` (the CLI/harness passes the resolved number); when `used > limit` set `result.budgetExceeded = true` and `result.budget = { used, limit, offenders }` where offenders is the selected rule ids; otherwise set `result.budgetExceeded = false` and `result.budget = { used, limit, offenders: [] }`. NEVER remove or truncate a selected rule (Pitfall 6). select() stays pure — it sets the flag; the CLI maps it to an exit code.
- `src/select/tokens.test.ts` — RED-first unit suite for estimateTokens (char/4 ceil boundaries) and the budget computation (a fixture whose selected summaries exceed a small budget trips `budgetExceeded` with offenders; a comfortable budget does not — no false trip).
- `src/select/eval-harness.ts` — exports the measurement functions: `scoreCase(result, expectedIds)` -> `{ tp, fp, fn }`; `runCases(index, cases)` -> per-case results; `aggregate(...)` -> `{ microRecall, microPrecision, recallBySeverity: { critical, high, medium, low } }`. Pure functions over `select()` output; recall is vacuously 1 when expected is empty, precision penalizes any selection on an empty-expected case (02-RESEARCH ties/empty scoring). Severity partition is on the EXPECTED rule's index severity (ground truth).
- `src/select/recall.test.ts` — the build-gating core-value suite: builds the eval corpus, runs every 02-01 case through `select()`, asserts `criticalRecall === 1.0` and `highRecall >= 0.9`, and REPORTS micro precision + per-severity recall via a console line (never asserted). This test failing fails the build.
- `src/select/select.property.test.ts` — fast-check invariants (`import * as fc`, numRuns 30, generate temp corpora via real buildIndex like 02-01/Phase 1): selected⊆triggered, stable/byte-identical ordering, total accounting, exclude-wins.
- `src/cli/commands/select.ts` — the `governance select` command: `parseArgs` (mirror build-index.ts) with `--index` (default `rule-index.json`), `--input` (signal JSON file; stdin if omitted), `--phase` (required), `--domains` (comma list -> string[]), `--budget` (number), `--format` (`json`|`text`, default `json`); `allowPositionals: false`. Reads + `validateSignal`s the signal, resolves budget (flag > config.json governance.token_budget > 2000), calls `select()`, writes result, `process.exit(1)` when `budgetExceeded`.
- `src/cli/index.ts` — MODIFIED: replace the `// Phase 2: case "select"` comment with `case "select": return (await import("./commands/select.js")).run(rest);` (lazy import, matching the established dispatch pattern).
- `src/cli/select.smoke.test.ts` — spawns the built CLI (`node dist/cli/index.js select ...`) against the eval index with a fixture signal: asserts a valid JSON SelectionResult on stdout and exit 0 for an in-budget run, and exit 1 + a stderr message for a `--budget 1` overflow run.
