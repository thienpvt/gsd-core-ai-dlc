<objective>
Close the CB-3 consent gate — the security load-bearer for the whole phase. The overlay's first activation in a project MUST record explicit consent via the loader's own verb (`gsd-tools capability install <spec> --scope project --yes`), never via aliasing or a hand-edited consent.json. This plan verifies the pre-consent (INACTIVE) and post-consent (ACTIVE) transition end-to-end through the loader itself, plus the tamper-detection content-binding that fails CLOSED if capability.json is mutated after consent.

This is Wave 3 because the manifest (04-01) must exist and validate, and the execute hook (04-02) must be in place for both gates' render-hooks to be meaningful post-consent. Files are distinct from 04-01/04-02 (no overlap).

This is the only non-autonomous plan in the phase: it requires a real human decision to grant consent (the CB-3 gate's whole point), so a `checkpoint:human-verify` task captures the actual grant. The pre/post render-hooks assertions are automated where possible.

Purpose: prove the overlay CANNOT activate without explicit user consent, and that tampering with the manifest after consent fails CLOSED. This is the security gate for the long-running loop — without it, any repo-plantable capability could execute arbitrary hooks on clone.
Output: a loader-driven consent integration test, a render-hooks verification script, and a runbook documenting the consent grant + revocation flow.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Repo-plantable `.gsd/capabilities/aidlc-governance/capability.json` -> loader (CB-3 gate) | A project-scope capability is UNTRUSTED repo content. The loader discovers it but treats it as INACTIVE until a user-owned consent record matches (realpath(projectRoot), id, bundleContentHash). This gate is the security load-bearer: without it, a clone/fork could execute arbitrary hook code with no user decision. |
| User-owned consent store (`GSD_HOME/.gsd/consent.json`) -> loader | The consent store lives OUTSIDE the repo (per-machine). The loader reads it via the bounded, fail-closed `readConsentStore`. A malformed/oversized store fails CLOSED (overlay stays INACTIVE). |
| Bundle content (capability.json + bundled artifacts) -> bundleContentHash -> consent record content binding | The consent record's `contentHash` binds the consent to the EXACT bundle content at grant time. Any mutation (a swapped capability.json, an edited hook script) changes the hash; the next load recomputes, fails to match, and the overlay flips INACTIVE — fail CLOSED on tamper. |
